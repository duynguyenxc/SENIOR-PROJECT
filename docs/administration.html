Administration & Maintenance

This section covers routine operational tasks that are not part of the day-to-day use described in the User Guide, what is needed to move the application to a new hosting provider, and suggestions for future enhancements.

Routine Operational Tasks

Updating the Weekly Menu

The weekly menu should be updated at the start of each week (or whenever the restaurant changes its offerings). Only the Super Admin can do this.

  1. Log in to the staff portal as Super Admin.
  2. Go to Menu Builder.
  3. Click each day that needs updating.
  4. Click Remove next to any dish that is no longer served on that day.
  5. Use the Add Dish to Menu form to add any new dishes for the week.

Changes take effect immediately — customers who reload the Menu page will see the updated menu.

Updating Takeout Sets

If a takeout set is temporarily unavailable (e.g., ingredients out of stock), toggle it to Unavailable from the Takeout Sets page. Unavailable sets are hidden from customers automatically. Toggle it back to Available when it returns to the menu.

If prices change, go to Takeout Sets, click Edit on the set, update the price, and save.

Managing Staff Accounts

When a new staff member joins: go to Manage Staff → Create Staff Account, enter their email and a temporary password, and click Create Staff. Share the credentials with them securely.

When a staff member leaves: click Disable next to their account. This immediately revokes their access without deleting the account record.

If a staff member forgets their password: click Reset Password, set a new temporary password, and share it with them.

Cleaning Up Abandoned Orders

Sometimes a customer starts the checkout process but closes the browser before completing payment. These orders remain in Pending status and never appear in the staff queue (only Paid orders are shown in the queue).

The system automatically cancels stale Pending orders when the Super Admin visits the Reports page. This triggers the cancel_stale_pending_orders() function, which marks orders as Cancelled if they have been in Pending status beyond a timeout threshold.

Recommended practice: the Super Admin should visit the Reports page at least once per week to trigger this cleanup automatically.

Managing Uploaded Images

Dish and takeout set images are stored in the web/uploads/ directory:

  • web/uploads/dishes/ — dish images uploaded via Menu Builder
  • web/uploads/takeout/ — takeout set images uploaded via Takeout Sets

Locally (Docker): the web/ folder is a mounted volume, so uploaded files persist between container restarts.

On Render (production): the filesystem is ephemeral — uploaded files are lost every time the server restarts or redeploys. See the Migration section for the recommended fix (persistent object storage).

Database Backup and Restore

Local Backup via phpMyAdmin

  1. Open phpMyAdmin at http://localhost:8081.
  2. Select the vegbuffet database from the left panel.
  3. Click the Export tab at the top of the page.
  4. Leave format as SQL and click Export.
  5. A .sql file is downloaded. Store it in a safe location.

Local Backup via Command Line

With the Docker containers running, run this command in a terminal:

docker exec senior-project-db-1 \
  mysqldump -u app -papppass vegbuffet > backup.sql

This saves a complete backup as backup.sql in your current directory.

Restoring from a Backup

docker exec -i senior-project-db-1 \
  mysql -u app -papppass vegbuffet < backup.sql

Production Database (TiDB Cloud)

TiDB Cloud provides automated backups from the cluster dashboard. To create a manual backup: log into tidbcloud.com → your cluster → Backup → Create Backup.

To export via command line, use mysqldump with the TiDB Cloud connection string found in the cluster dashboard. The connection requires SSL and a specific port (typically 4000).

Moving to a New Hosting Provider

Current Hosting Setup

DatabaseTiDB Cloud (MySQL-compatible serverless)Free tier

⚠ Free tier limitations
  • Render free tier — the web server spins down after 15 minutes of inactivity. The first request after inactivity takes 30–60 seconds to load.
  • TiDB Cloud free tier — the database hibernates after 7 days without activity. Free storage is capped at 5 GB.
For a production environment, upgrade both to paid tiers or migrate to alternative providers.

What Is Needed to Migrate

Because the application is fully containerized with Docker, it can be moved to any hosting provider that supports Docker containers: AWS ECS, DigitalOcean App Platform, Railway, Fly.io, a self-managed VPS, etc.

Web Server Requirements on the New Host

  • Docker support (ability to run a custom Docker image)
  • Ability to set environment variables
  • Port 80 or 443 exposed to the public
  • Persistent storage volume for uploaded images (or external object storage — see below)

Database Requirements on the New Host

  • Any MySQL 8.0+ or MariaDB 10.6+ compatible database server
  • The schema from db/init/01_schema.sql must be applied to the new database
  • Optionally import a data backup from the old database

Step-by-Step Migration Guide

1

Export Data from TiDB Cloud (Current Database)

Using a MySQL client, connect to TiDB Cloud and export:

mysqldump \
  --host=<tidb-host> \
  --port=4000 \
  --user=<username> \
  --password \
  --ssl-mode=REQUIRED \
  vegbuffet > full-backup.sql
2

Create the Database on the New Host

Create a database named vegbuffet (or any name; update the env var DB_NAME accordingly).

Import the backup:

mysql -h <new-db-host> -u <user> -p vegbuffet < full-backup.sql

If starting fresh (no data to migrate), just apply the schema and seed files:

mysql -h <new-db-host> -u <user> -p vegbuffet < db/init/01_schema.sql
mysql -h <new-db-host> -u <user> -p vegbuffet < db/init/02_seed.sql
3

Deploy the Application Container

Deploy the Docker image from web/Dockerfile to the new hosting platform. Set the following environment variables on the new host: 

DB_HOST=<new-database-host>
DB_PORT=3306
DB_NAME=vegbuffet
DB_USER=<db-username>
DB_PASS=<db-password>
APP_URL=https://<your-new-domain>
STRIPE_SECRET_KEY=sk_test_...
STRIPE_PUBLISHABLE_KEY=pk_test_...

If the new database requires SSL (like TiDB Cloud), also set: 

DB_SSL_CA_PATH=/etc/ssl/certs/ca-certificates.crt
DB_SSL_VERIFY_SERVER_CERT=true
4

Update APP_URL

Set APP_URL to the new public URL of the application. This value is used to build Stripe's redirect URL after payment. If it is wrong, customers will be sent back to the old domain after paying.

5

Set Up Persistent Image Storage (Strongly Recommended)

On Render (and most Docker cloud platforms), the local filesystem is ephemeral — uploaded files disappear when the container restarts.

The recommended solution is to use object storage such as:

  • Amazon S3
  • Cloudflare R2 (no egress fees on free tier)

This requires modifying web/lib/upload.php to upload files to the bucket instead of the local filesystem, and updating stored imageUrl values to point to the bucket's public URL.

As a simpler temporary workaround: always re-upload images after a server restart using the admin interface.

Enhancing and Extending the Project

The following improvements are not currently implemented but represent logical next steps for taking this project further in a real production environment.

1. Switch Stripe to Live Mode (Real Payments)

The current setup uses Stripe in test/sandbox mode. To accept real payments:

  1. Complete Stripe's business verification process (business name, address, bank account for payouts).
  2. Replace the sk_test_ / pk_test_ keys with live sk_live_ / pk_live_ keys in the environment variables.
  3. Optionally configure Stripe webhooks for more reliable payment event handling.

2. Persistent Image Storage (S3 / Cloudflare R2)

As described above, move uploaded images to external object storage so they survive server restarts and redeployments. This is the most impactful near-term infrastructure change.

3. Customer Password Self-Service

Currently, customers cannot reset their own password if they forget it. Adding a "Forgot Password" flow (email with a one-time reset link) would improve usability and security. This requires integrating a transactional email service (e.g., SendGrid, Postmark, or AWS SES).

4. Order Confirmation and Status Emails

Automatically emailing customers when:

  • Their order is confirmed (after payment)
  • Their order is marked Ready for Pickup

This eliminates the need for customers to keep the browser open to track status. Implementation: integrate an email API in web/lib/ and call it from within the order status update logic in web/lib/orders.php.

5. Automatic Refunds on Cancellation

When a Super Admin cancels an order that has already been paid, the system currently only changes the order status. A future enhancement would automatically call the Stripe Refunds API to issue a refund at the same time, reducing manual steps and the risk of forgetting to refund the customer.

6. Export Reports to CSV or PDF

Adding an export button to the Reports page would let the restaurant owner download sales data without logging into the admin interface. This could be implemented by adding a PHP endpoint that queries the same data as the Reports page and outputs it as a downloadable file.

7. Dish Availability Toggle Per Day

Currently, a dish can only be added or removed from a day's menu entirely. A per-day availability toggle (e.g., "sold out today") would let staff make quick day-of adjustments without permanently removing a dish from the weekly menu.

8. Push Notifications for New Orders

The order queue currently uses Server-Sent Events (SSE) to refresh in real time, but requires the staff to be actively looking at the dashboard. Browser push notifications (or an audio alert) would immediately notify staff of new orders even when they are on another browser tab.

9. Discount and Promo Code Support

Stripe Checkout natively supports coupon codes. The Super Admin could create Stripe coupons from the admin interface, and customers could enter a promo code during checkout.

10. Multi-Language Support

Adding language options (e.g., English and Vietnamese) using PHP's localization functions would improve accessibility for customers who are not comfortable reading English.
